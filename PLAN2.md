# تقرير شامل: الأدوات والمهارات المقترحة لتطبيق فرق الوكلاء متعددة النماذج

## ملخص تنفيذي

يستهدف هذا التقرير تحليل خطة تطبيق بناء فرق وكلاء متعددة النماذج (TypeScript + LangGraph + LiteLLM) وتحديد أفضل الأدوات (Tools)، والمهارات (Skills)، والإضافات المعمارية التي ترفع كفاءة التطبيق الفعلية. يعتمد التحليل على أحدث المعطيات المتاحة، مع التركيز على ما يتكامل مباشرة مع المعمارية المُعتمَدة (LangGraph + LiteLLM + Fastify + Next.js).

## القسم الثالث: المهارات (Agent Skills) الموصى بإضافتها

### مفهوم Agent Skills

Agent Skills هي مجلدات تحتوي ملف `SKILL.md` مع تعليمات وبيانات وصفية تُعلّم الوكيل كيفية تنفيذ مهام متخصصة. تُحمَّل عند الطلب فقط (Progressive Disclosure) مما يحافظ على كفاءة نافذة السياق.[14][21][22]

البنية النموذجية:

```
skill-name/
├── SKILL.md          # مطلوب: تعليمات + بيانات وصفية
├── scripts/          # اختياري: سكربتات مساعدة
├── templates/        # اختياري: قوالب
└── references/       # اختياري: مراجع
```

## القسم الرابع: إضافات معمارية مقترحة

### 1. دعم بروتوكول A2A (Agent-to-Agent)

بروتوكول A2A من Google يسمح لوكلاء مبنيين على أُطر مختلفة (LangGraph, CrewAI, Semantic Kernel) بالتواصل والتنسيق عبر واجهة موحدة. هذا يُمكّن التطبيق من:[25][26]

- استقبال طلبات من وكلاء خارجيين عبر بطاقة وكيل (Agent Card) موحدة
- إرسال مهام لوكلاء متخصصين خارجيين عند الحاجة
- التشغيل البيني مع أكثر من 50 شريك تقني يدعم البروتوكول[25]

LiteLLM يدعم بالفعل بوابة A2A Gateway التي تسمح بتسجيل واختبار وكلاء LangGraph عبر واجهة مرئية.[27]

### 2. نظام مهارات مُدمَج (Built-in Skills Engine)

بدلاً من تثبيت المهارات يدويًا، يُقترح بناء محرك مهارات داخلي:

- **SkillRegistry**: اكتشاف وتحميل وإدارة مهارات من مجلد `skills/`
- **SkillActivator**: حقن تعليمات المهارة المناسبة في سياق المحادثة عند الطلب
- **SkillWatcher**: مراقبة مجلد المهارات وإعادة تحميل أي تغيير تلقائيًا[21]

هذا النهج (Progressive Disclosure) يضمن أن الوكيل يعرف المهارات المتاحة (100 توكن لكل مهارة) لكنه يُحمّل التفاصيل فقط عند التفعيل.[14][21]

### 3. ذاكرة طويلة المدى (Long-term Memory)

إضافة طبقة ذاكرة متجهية تتيح للوكلاء:

- تذكر نتائج مهام سابقة واسترجاعها عند الحاجة
- استخدام `load_memory_tool` للبحث في المعرفة المتراكمة[28]
- استخدام `preload_memory_tool` لتحميل سياق مسبق قبل استدعاء النموذج[28]

Qdrant أو LanceDB خيارات مناسبة لمحرك البحث المتجهي مع تكامل LangGraph.[10][29]

### 4. نظام Guardrails مُدمَج

طبقة حماية تعمل في الوقت الحقيقي:

- التحقق من مدخلات ومخرجات كل وكيل
- كشف حقن البرومبت (Prompt Injection) والمحتوى السام
- فحص PII (بيانات شخصية) قبل الإرسال لنماذج خارجية
- إعادة المحاولة تلقائيًا عند فشل التحقق[20][19]

```
Guardrails Layer → Input Guard → Model Call → Output Guard → Response
```

### 5. استدعاء أدوات برمجي (Programmatic Tool Calling)

بدلاً من استدعاء كل أداة على حدة (ما يستهلك توكنات)، يكتب الوكيل سكربت يستدعي الأدوات في حلقات ودُفعات:

- تحسين كفاءة التوكنات بشكل كبير (الوكيل يقرأ مخرجات السكربت فقط، وليس 50 استدعاء وسيط)[30]
- تقليل زمن الاستجابة
- مناسب للوكلاء المتخصصين الذين يحتاجون عمليات متكررة

### 6. نظام ملفات وسيط (Filesystem-mediated Context)

تمثيل البيانات المعقدة كملفات نصية قابلة للبحث بدل تحميلها في نافذة السياق:

- مثال: تحويل طلب سحب GitHub إلى `./pr/<id>/change.patch` و`./pr/<id>/metadata.json`[30]
- الوكيل يستخدم `grep` للبحث بدل تحميل كل شيء
- يُمكن اعتراض عمليات القراءة لتنفيذ "قراءات عمياء" (Blind Reads) تُملأ بالمحتوى المطلوب ديناميكيًا[30]

### 7. تسلسل هرمي للأدوات (Bigtool Library)

مكتبة LangGraph Bigtool تُطبّق بحث تشابه دلالي على أوصاف الأدوات لاختيار الأنسب لكل مهمة. عند وجود عدد كبير من الأدوات:[31]

- لا حاجة لإرفاق جميع الأدوات في كل استدعاء
- البحث الدلالي يختار الأدوات ذات الصلة فقط
- تحسين دقة اختيار الأدوات بنسبة تصل إلى 3 أضعاف[31]

### 8. تحسين الأداء عبر تخزين السياق المؤقت (Context Caching)

تقسيم نافذة السياق إلى منطقتين:

- **منطقة ثابتة** (أمام النافذة): تعليمات النظام، تعريفات الأدوار، الأدوات المتاحة – تستفيد من Prefix Caching
- **منطقة ديناميكية** (آخر النافذة): رسائل المحادثة الجارية والنتائج[28]

هذا يُقلل زمن الاستدلال ويخفض التكلفة عند الاستدعاءات المتتالية لنفس الفريق.

## القسم الخامس: إضافات مُقترحة لتدفق العمل

### 1. عقدة Human Feedback Loop

إضافة عقدة في الرسم البياني بعد `verifierNode` تسمح بجمع تقييم المستخدم النهائي:

- تقييم من 1 إلى 5 نجوم
- ملاحظات نصية حرة
- تخزين التقييمات في `run_events` لتحسين مستقبلي

### 2. عقدة Cost Tracking

على الرغم من سياسة "الجودة أولاً"، يُفيد تتبع التكلفة الفعلية لكل تشغيل:

- تسجيل توكنات المدخلات/المخرجات لكل نموذج
- حساب التكلفة التقريبية عبر أسعار LiteLLM
- عرض التكلفة في التقرير النهائي دون أن تؤثر على قرار الاختيار

### 3. عقدة Artifact Manager

إدارة مخرجات الوكلاء بشكل موحد:

- حفظ ملفات الكود والتقارير والبيانات في `artifacts` مع بيانات وصفية
- إتاحة تحميل المخرجات لاحقًا عبر API
- ربط كل مخرج بالوكيل والنموذج الذي أنتجه

### 4. نظام Template Marketplace

بدلاً من 4 قوالب ثابتة (Research, Coding, Content, Data Analysis):

- السماح بإنشاء قوالب مخصصة عبر ملفات YAML
- تبادل القوالب عبر مستودع مركزي
- دعم قوالب مُركّبة (Composite Templates) تجمع أدوارًا من قوالب متعددة
